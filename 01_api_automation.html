📡 บทที่ 1: API Automation

บทนี้จะสอนตั้งแต่ API คืออะไร จนถึงการเขียน test อัตโนมัติด้วย Python และ Postman

🍽️ API คืออะไร?

ลองนึกภาพร้านอาหาร เราไม่ได้เดินเข้าครัวเองเพื่อหยิบอาหาร แต่สั่งผ่านพนักงานเสิร์ฟ


  แอปมือถือ / เว็บ          ←→         API          ←→       Server / Database
  (ลูกค้า)                         (พนักงานเสิร์ฟ)              (ครัว)

  ตัวอย่าง:
  แอปสั่งอาหาร  →  ส่งคำขอ "ขอเมนูทั้งหมด"  →  API ดึงข้อมูลจาก DB แล้วส่งกลับมา
สรุปง่ายๆ: API คือ "ตัวกลาง" ที่รับคำขอจากแอป แล้วไปดึงข้อมูลจาก server กลับมาให้
เราในฐานะ QA ต้องทดสอบว่า API ทำงานถูกต้องมั้ย ตอบกลับถูกมั้ย

📬 HTTP Method คืออะไร?

เวลาเราคุยกับ API เราต้องบอกว่าจะทำอะไร โดยใช้ HTTP Method เหมือนกับการบอกพนักงานเสิร์ฟว่าจะสั่งอะไร

Methodทำอะไรตัวอย่างในชีวิตจริง
GETขอดูข้อมูลดูรายการสินค้า, ดูโปรไฟล์ user
POSTสร้างข้อมูลใหม่สมัครสมาชิก, โพสต์รูป
PUTแก้ไขข้อมูลทั้งหมดแก้ข้อมูลโปรไฟล์ทั้งหมด
PATCHแก้ไขข้อมูลบางส่วนแก้แค่ชื่อ ไม่แก้อย่างอื่น
DELETEลบข้อมูลลบ account, ลบโพสต์

🔢 Status Code คืออะไร?

เวลา API ตอบกลับมา มันจะบอกผลด้วยตัวเลข เหมือนพนักงานเสิร์ฟบอกว่า "ได้เลย" หรือ "ของหมดแล้ว"

Codeความหมายเจอเมื่อไหร่
200 OKสำเร็จGET ข้อมูลสำเร็จ
201 Createdสร้างสำเร็จPOST สร้าง user ใหม่สำเร็จ
400 Bad Requestข้อมูลที่ส่งไปผิดกรอก email ผิดรูปแบบ
401 Unauthorizedยังไม่ได้ loginเรียก API โดยไม่มี token
403 Forbiddenไม่มีสิทธิ์user ธรรมดาเข้า admin page
404 Not Foundไม่เจอข้อมูลหา user id ที่ไม่มีในระบบ
500 Server Errorserver พังbug ใน code ของ developer

🧪 ทดสอบ API ด้วย Postman (Manual)

Postman คือ tool ที่ใช้ทดสอบ API แบบ manual ก่อนที่จะเขียน automation

ขั้นตอนใช้ Postman

1 โหลด Postman — ไปที่ postman.com/downloads แล้วโหลดฟรี
2 สร้าง Request ใหม่ — กด "New" → "HTTP Request"
3 ใส่ข้อมูล 
Method:  GET
URL:     https://jsonplaceholder.typicode.com/users/1
กด Send
4 ดูผลที่ได้ 
{
  "id": 1,
  "name": "Leanne Graham",
  "username": "Bret",
  "email": "Sincere@april.biz",
  "phone": "1-770-736-0988 x56442"
}

เขียน Test ใน Postman (Automate บางส่วน)

Postman มี tab "Tests" ที่เราเขียน JavaScript เพื่อเช็คผลอัตโนมัติได้

// ===== Tests tab ใน Postman =====

// Test 1: เช็ค status code
pm.test("Status code is 200", function () {
    // pm.response.to.have.status = เช็ค status code
    pm.response.to.have.status(200);
});

// Test 2: เช็คว่า response มี field "name"
pm.test("Response has name field", function () {
    // pm.response.json() = แปลง response เป็น object
    var jsonData = pm.response.json();

    // pm.expect = เช็คเงื่อนไข
    pm.expect(jsonData.name).to.exist;
});

// Test 3: เช็คว่า response time ไม่เกิน 500ms
pm.test("Response time is less than 500ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(500);
});

// Test 4: เช็คว่า Content-Type เป็น JSON
pm.test("Content-Type is JSON", function () {
    pm.response.to.have.header("Content-Type");
    pm.expect(pm.response.headers.get("Content-Type")).to.include("application/json");
});

Postman Collection Runner (รัน test หลายอันพร้อมกัน)

Collection คือกลุ่มของ request ที่เกี่ยวข้องกัน เช่น "User API" มี GET, POST, DELETE

Collection Runner คือการรัน request ทั้งหมดใน Collection พร้อมกันทีเดียว แล้วดูผลรวม 
วิธีใช้ Collection Runner:
1. สร้าง Collection ชื่อ "User API Tests"
2. เพิ่ม request ทั้งหมดเข้าไป
3. กด "Run Collection"
4. ดูผลว่า test ไหน pass/fail

Newman (รัน Postman ใน Terminal)

Newman คือ CLI ของ Postman ทำให้รัน Collection ใน Terminal หรือ CI/CD ได้

# ติดตั้ง Newman
npm install -g newman

# Export Collection จาก Postman แล้วรัน
newman run my-collection.json

# รันแล้วได้ HTML report
newman run my-collection.json --reporters html --reporter-html-export report.html

🐍 ทดสอบ API ด้วย Python (Full Automation)

Python + pytest คือวิธีที่ยืดหยุ่นกว่า Postman เหมาะสำหรับรันใน CI/CD pipeline

Manual vs Automation เปรียบเทียบ

Postman (Manual)Python (Automation)
พิมพ์ URL แล้วกด Sendrequests.get("URL")
ดู status code ด้วยตาassert response.status_code == 200
ดู JSON body ด้วยตาassert "name" in response.json()
ทำซ้ำเองทุกครั้งรันอัตโนมัติได้เรื่อยๆ
รัน CI/CD ได้ด้วย Newmanรัน CI/CD ได้ด้วย pytest

โครงสร้าง API Test มี 2 ส่วน

ส่วนที่ 1  →  Import library + ตั้งค่า BASE_URL
ส่วนที่ 2  →  Test Cases จริงๆ (แต่ละ function = 1 test case)

สร้างไฟล์ api_tests/test_api.py

# ============================================================
# ส่วนที่ 1: Import library + ตั้งค่า
# ============================================================

# requests คือ library สำหรับยิง HTTP request
# เหมือน Postman แต่เป็น code
import requests

# BASE_URL คือ URL หลักของ API ที่จะทดสอบ
# เก็บไว้ที่เดียว ถ้าต้องเปลี่ยนก็แก้แค่ที่นี่ที่เดียว
# jsonplaceholder.typicode.com = API ทดสอบฟรี ไม่ต้อง login
BASE_URL = "https://jsonplaceholder.typicode.com"


# ============================================================
# ส่วนที่ 2: Test Cases
# ทุก function ที่ขึ้นต้นด้วย test_ จะถูก pytest รันเป็น test case
# ============================================================


# ----------------------------------------------------------
# Test 1: GET user — เช็คว่าดึงข้อมูล user ได้มั้ย
# ----------------------------------------------------------
def test_get_user_returns_200():
    # requests.get = ส่ง GET request ไปที่ URL นี้
    # เหมือนกับพิมพ์ URL ใน Postman แล้วกด Send
    response = requests.get(f"{BASE_URL}/users/1")

    # assert = เช็คเงื่อนไข
    # ถ้าเงื่อนไขเป็นจริง → ผ่าน อ่านบรรทัดต่อไป
    # ถ้าเงื่อนไขเป็นเท็จ → test fail ทันที แสดง error
    assert response.status_code == 200


# ----------------------------------------------------------
# Test 2: GET user — เช็คว่า response มี field ครบมั้ย
# ----------------------------------------------------------
def test_get_user_has_correct_fields():
    response = requests.get(f"{BASE_URL}/users/1")

    # .json() = แปลง response จาก text เป็น dictionary Python
    # เหมือนข้อมูล JSON ที่เห็นใน Postman panel ด้านล่าง
    body = response.json()

    # "id" in body = เช็คว่า key "id" มีอยู่ใน dictionary มั้ย
    assert "id" in body       # ต้องมี field id
    assert "name" in body     # ต้องมี field name
    assert "email" in body    # ต้องมี field email
    assert "phone" in body    # ต้องมี field phone


# ----------------------------------------------------------
# Test 3: GET user — เช็คว่าค่าที่ได้ถูกต้องมั้ย
# ----------------------------------------------------------
def test_get_user_correct_values():
    response = requests.get(f"{BASE_URL}/users/1")
    body = response.json()

    # เช็คว่า id ที่ได้กลับมาตรงกับที่ขอมั้ย
    assert body["id"] == 1

    # เช็คว่า name ไม่ใช่ค่าว่าง
    assert body["name"] != ""

    # เช็คว่า email มี @ อยู่ (รูปแบบ email ถูกต้อง)
    assert "@" in body["email"]


# ----------------------------------------------------------
# Test 4: POST — เช็คว่าสร้างข้อมูลใหม่ได้มั้ย
# ----------------------------------------------------------
def test_create_post_returns_201():
    # payload = ข้อมูลที่จะส่งไปสร้างใหม่
    # เหมือนกับที่เราใส่ใน Body tab ของ Postman
    payload = {
        "title": "test post",
        "body": "hello world",
        "userId": 1
    }

    # requests.post = ส่ง POST request พร้อมข้อมูล
    # json=payload = ส่ง payload เป็น JSON body
    response = requests.post(f"{BASE_URL}/posts", json=payload)

    # 201 = Created = สร้างสำเร็จ
    assert response.status_code == 201

    # เช็คว่าข้อมูลที่ได้กลับมาตรงกับที่ส่งไปมั้ย
    result = response.json()
    assert result["title"] == "test post"
    assert result["body"] == "hello world"
    assert result["userId"] == 1


# ----------------------------------------------------------
# Test 5: PUT — เช็คว่าแก้ไขข้อมูลได้มั้ย
# ----------------------------------------------------------
def test_update_post_returns_200():
    # ข้อมูลที่จะแก้ไข
    payload = {
        "id": 1,
        "title": "updated title",
        "body": "updated body",
        "userId": 1
    }

    # requests.put = ส่ง PUT request เพื่อแก้ไขข้อมูล
    response = requests.put(f"{BASE_URL}/posts/1", json=payload)

    assert response.status_code == 200
    assert response.json()["title"] == "updated title"


# ----------------------------------------------------------
# Test 6: DELETE — เช็คว่าลบข้อมูลได้มั้ย
# ----------------------------------------------------------
def test_delete_post_returns_200():
    # requests.delete = ส่ง DELETE request
    response = requests.delete(f"{BASE_URL}/posts/1")

    # 200 = OK = ลบสำเร็จ
    assert response.status_code == 200


# ----------------------------------------------------------
# Test 7: Edge Case — หา user ที่ไม่มีในระบบ ต้องได้ 404
# ----------------------------------------------------------
def test_get_nonexistent_user_returns_404():
    # ส่ง id ที่ไม่มีในระบบ
    response = requests.get(f"{BASE_URL}/users/99999")

    # ต้องได้ 404 Not Found กลับมา
    assert response.status_code == 404

📝 Postman Syntax พื้นฐานที่ควรรู้

syntax เหล่านี้คือสิ่งที่ใช้บ่อยที่สุดใน Postman Tests tab และเหมาะสำหรับใช้สัมภาษณ์ candidate

1. เช็ค Status Code

pm.test("Status code is 200", function () {
    // pm.test("ชื่อ test", function(){...})
    // ชื่อ test = จะแสดงใน report ตั้งชื่ออะไรก็ได้ให้สื่อความหมาย
    pm.response.to.have.status(200);
});

2. เช็คว่า field มีอยู่มั้ย (.to.exist)

pm.test("Response has name field", function () {
    // pm.response.json() = แปลง response จาก text ธรรมดา → เป็น object ที่จับต้องได้
    // เหมือนแปลงกล่องปิดตา → กล่องที่เปิดดูข้างในได้
    var jsonData = pm.response.json();

    // jsonData.name = ดึงค่าของ field "name" ออกมา
    // .to.exist = เช็คว่ามีค่าอยู่จริงมั้ย ไม่ใช่ค่าว่าง
    // ถ้าไม่มี field name → test fail ทันที
    pm.expect(jsonData.name).to.exist;
});
ตัวอย่างให้เห็นภาพ:
สมมุติ API ส่งกลับมาแบบนี้ 
{
  "id": 1,
  "name": "สมชาย",
  "email": "somchai@test.com"
}
หลังจาก .json() แล้ว เราเข้าถึงได้เลยว่า
jsonData.name = "สมชาย"
jsonData.email = "somchai@test.com"

ถ้า API ไม่มี field name ส่งกลับมา → jsonData.name = undefined → test fail ทันที

3. เช็คว่าค่าถูกต้องมั้ย (.to.eql)

pm.test("Name is correct", function () {
    var jsonData = pm.response.json();

    // .to.eql = เช็คว่าค่าตรงกับที่กำหนดมั้ย
    // ต่างจาก .to.exist ตรงที่ระบุค่าที่คาดหวังได้เลย
    pm.expect(jsonData.name).to.eql("สมชาย");
});
Syntaxใช้เมื่อไหร่ตัวอย่าง
.to.existเช็คแค่ว่ามี field อยู่มั้ย ไม่สนใจค่ามี field name มั้ย?
.to.eqlเช็คว่าค่าตรงกับที่กำหนดมั้ยname ต้องเป็น "สมชาย"

4. เช็ค Response Time

pm.test("Response time is acceptable", function () {
    // pm.response.responseTime = เวลาที่ API ใช้ตอบกลับ (หน่วยเป็น ms)
    // .to.be.below(500) = ต้องน้อยกว่า 500ms
    pm.expect(pm.response.responseTime).to.be.below(500);
});

5. เช็ค Header

pm.test("Content-Type is JSON", function () {
    // Header คือข้อมูลเพิ่มเติมที่ API ส่งมาพร้อม response
    // Content-Type บอกว่าข้อมูลที่ส่งมาเป็นประเภทอะไร
    pm.expect(pm.response.headers.get("Content-Type")).to.include("application/json");
});

สรุป Pattern ที่ต้องจำ

// โครงสร้างหลักของทุก test ใน Postman
pm.test("ชื่อ test ที่จะโชว์ใน report", function () {

    var jsonData = pm.response.json();  // แปลง response เป็น object ก่อนเสมอ

    pm.response.to.have.status(200);                        // เช็ค status code
    pm.expect(jsonData.name).to.exist;                      // เช็คว่ามี field มั้ย
    pm.expect(jsonData.name).to.eql("ค่าที่คาดหวัง");       // เช็คว่าค่าถูกมั้ย
    pm.expect(pm.response.responseTime).to.be.below(500);  // เช็ค response time

});

❓ คำถามสัมภาษณ์ที่ควรรู้

ระดับเริ่มต้น

Q: pm.test คืออะไร?
A: function สำหรับสร้าง test case ใน Postman รับ 2 อย่างคือ ชื่อ test และสิ่งที่จะเช็ค

Q: pm.response.json() คืออะไร?
A: แปลง response จาก text ธรรมดาให้เป็น object เพื่อให้เข้าถึง field แต่ละตัวได้

Q: .to.exist กับ .to.eql ต่างกันยังไง?
A: exist = เช็คแค่ว่ามี field อยู่มั้ย, eql = เช็คว่าค่าตรงกับที่กำหนดมั้ย

Q: Status Code 200, 201, 400, 401, 404, 500 คืออะไร?
A: 200=สำเร็จ, 201=สร้างสำเร็จ, 400=ข้อมูลผิด, 401=ไม่ได้ login, 404=ไม่เจอ, 500=server พัง

ระดับกลาง

Q: นอกจาก status code ควรเช็คอะไรอีก?
A: response body (field ครบมั้ย ค่าถูกมั้ย), response time, header

Q: Edge Case คืออะไร ยกตัวอย่างได้มั้ย?
A: กรณีผิดปกติ เช่น ส่งข้อมูลว่าง ส่ง id ที่ไม่มีในระบบ เพื่อเช็คว่า API จัดการ error ได้ถูกต้องมั้ย

Q: Collection Runner คืออะไร?
A: รัน request ทั้งหมดใน Collection พร้อมกันทีเดียว แล้วดูผลรวมว่า test ไหน pass/fail

Q: Newman คืออะไร?
A: CLI ของ Postman ทำให้รัน Collection ใน Terminal หรือ CI/CD ได้โดยไม่ต้องเปิด Postman

❓ คำถามสัมภาษณ์ที่ควรรู้

Q: Assert คืออะไร?
A: การตรวจสอบว่าผลลัพธ์ตรงกับที่คาดหวังมั้ย ถ้าไม่ตรง test fail ทันที

Q: ทำไมต้อง Automate API แทนที่จะใช้ Postman?
A: Postman ต้องนั่งกดเองทุกครั้ง Automation รันซ้ำได้เองและรันใน CI/CD ได้

Q: Postman กับ Python ต่างกันยังไง?
A: Postman ใช้งานง่ายกว่า เหมาะ manual test, Python ยืดหยุ่นกว่า เหมาะ CI/CD

Q: Status Code 200, 201, 400, 401, 404, 500 คืออะไร?
A: 200=สำเร็จ, 201=สร้างสำเร็จ, 400=ข้อมูลผิด, 401=ไม่ได้ login, 404=ไม่เจอ, 500=server พัง

Q: Edge Case คืออะไร?
A: การทดสอบกรณีที่ผิดปกติ เช่น ส่งข้อมูลว่าง ส่ง id ที่ไม่มี เพื่อเช็คว่า API จัดการ error ได้ถูกต้องมั้ย
