บทที่ 4: CI/CD Pipeline กับ Azure DevOps

บทนี้แบ่งเป็น 2 ส่วน กดเลือกได้เลย

ก่อนอื่น ลองนึกภาพนี้ก่อนเลย

สมมุติเราทำงานเป็น QA ในบริษัท มีทีม developer 5 คน แต่ละคน push code ขึ้นวันละหลายครั้ง

ถ้าไม่มี CI/CD เกิดอะไรขึ้น?

Developer คนที่ 1 แก้ code แล้ว push ขึ้น
Developer คนที่ 2 แก้ code แล้ว push ขึ้น
Developer คนที่ 3 แก้ code แล้ว push ขึ้น

QA ต้องนั่งรัน test เองทุกครั้ง วันนึงอาจต้องรัน 10-20 ครั้ง
ถ้าลืมรัน หรือรันไม่ทัน → bug หลุดไป production → ลูกค้าเจอ bug
ถ้ามี CI/CD เกิดอะไรขึ้น?

Developer คนที่ 1 แก้ code แล้ว push ขึ้น → ระบบรัน test อัตโนมัติทันที
Developer คนที่ 2 แก้ code แล้ว push ขึ้น → ระบบรัน test อัตโนมัติทันที
Developer คนที่ 3 แก้ code แล้ว push ขึ้น → ระบบรัน test อัตโนมัติทันที

QA ไม่ต้องนั่งรัน test เอง ระบบทำให้หมด
ถ้า test fail → แจ้งเตือนทันที → แก้ก่อน merge ได้

CI/CD คืออะไร? แยกทีละคำ

หลายคนได้ยินคำนี้แล้วงง เพราะมันเป็นคำย่อ ลองแยกดูทีละคำ

CI = Continuous Integration
แปลตรงๆ คือ "การรวม code อย่างต่อเนื่อง"
ความหมายจริงๆ คือ ทุกครั้งที่มีการ push code ขึ้นมา ระบบจะรัน test อัตโนมัติทันที
เพื่อเช็คว่า code ใหม่ที่เพิ่งเข้ามาทำให้อะไรพังมั้ย

CD = Continuous Delivery
แปลตรงๆ คือ "การส่ง code อย่างต่อเนื่อง"
ความหมายจริงๆ คือ หลังจาก test ผ่านแล้ว ระบบจะ deploy code ขึ้น server อัตโนมัติ
โดยไม่ต้องให้ใครมานั่งกด deploy เอง
ในฐานะ QA เราสนใจแค่ CI เป็นหลัก
คือการที่ระบบรัน test script ของเราอัตโนมัติทุกครั้งที่มีการ push code

Pipeline คืออะไร? ต่างจาก CI/CD ยังไง?

นี่คือจุดที่หลายคนสับสน ขอแยกให้ชัดเลย

CI/CD คือ "แนวคิด" — เหมือนการบอกว่า "อยากให้โรงงานผลิตรถยนต์มีระบบตรวจสอบคุณภาพอัตโนมัติ"

Pipeline คือ "วิธีทำให้แนวคิดนั้นเกิดขึ้นจริง" — เหมือนการออกแบบสายพานการผลิตจริงๆ ว่ามีกี่ขั้นตอน แต่ละขั้นทำอะไร

พูดง่ายๆ คือ Pipeline คือไฟล์ที่เราเขียนบอกว่า "เมื่อมี code ใหม่เข้ามา ให้ทำอะไรบ้าง เรียงตามลำดับ" 
ตัวอย่าง Pipeline ที่เราจะสร้าง:

เมื่อมีการ push code ขึ้น branch main
        ↓
ขั้นตอนที่ 1: เปิดเครื่อง Linux เสมือนขึ้นมา
        ↓
ขั้นตอนที่ 2: ติดตั้ง Python version 3.11
        ↓
ขั้นตอนที่ 3: ติดตั้ง library ทั้งหมดจาก requirements.txt
        ↓
ขั้นตอนที่ 4: รัน Playwright test
        ↓
ขั้นตอนที่ 5: รัน API test
        ↓
ขั้นตอนที่ 6: รัน Robot Framework test
        ↓
ขั้นตอนที่ 7: เก็บ report ไว้ให้ดาวน์โหลด
        ↓
แจ้งผลว่า pass หรือ fail

Azure DevOps คืออะไร? ทำไมต้องใช้?

Azure DevOps คือ platform ของ Microsoft ที่รวมทุกอย่างที่ทีม software ต้องการไว้ในที่เดียว

ส่วนคืออะไรเปรียบเหมือน
Reposที่เก็บ code ทั้งหมดบน cloudGoogle Drive แต่สำหรับ code โดยเฉพาะ
Pipelinesรัน test และ deploy อัตโนมัติสายพานโรงงานที่ทำงานเองอัตโนมัติ
Boardsจัดการงาน task และ bugJira หรือ Trello
Artifactsเก็บผลลัพธ์ เช่น report หลัง test รันที่เก็บของที่ผลิตได้จากโรงงาน
Test Plansจัดการ test caseสมุดบันทึก test case
เราจะใช้แค่ 3 ส่วนหลัก:
Repos → เก็บ code และ script test ของเรา
Pipelines → รัน test อัตโนมัติทุกครั้งที่ push code
Artifacts → เก็บ report ที่ได้หลัง test รันเสร็จ

Repo คืออะไร? ทำไมต้องเซต?

ก่อนจะตั้ง Pipeline ได้ ต้องมี Repo ก่อน เพราะ Pipeline ต้องดึง code จาก Repo มารัน

Repo = ที่เก็บ code บน cloud

ลองนึกภาพว่าเราเขียน test script ไว้บนเครื่องเรา แต่ Pipeline อยู่บน Azure DevOps
Pipeline ไม่รู้ว่า code เราอยู่ที่ไหน ถ้าไม่มี Repo

เราต้อง push code ขึ้น Repo ก่อน แล้ว Pipeline ถึงจะดึงไปรันได้
คำศัพท์ความหมายเปรียบเหมือน
Repository (Repo)ที่เก็บ code ทั้งหมดบน cloudGoogle Drive สำหรับ code
Branchสาขาของ code เช่น main, developสำเนาของ code ที่แยกออกมาทำงานแยกกัน
Commitบันทึก snapshot ของ code พร้อมข้อความกด Save พร้อมเขียนว่าแก้อะไร
Pushส่ง code จากเครื่องเราขึ้น Repoอัปโหลดขึ้น cloud
Pullดึง code จาก Repo มาที่เครื่องเราดาวน์โหลดจาก cloud
Pull Request (PR)ขอ merge code จาก branch นึงไปอีก branchส่งงานให้หัวหน้าตรวจก่อน merge

azure-pipelines.yml คืออะไร? ทำไมต้องมีไฟล์นี้?

ไฟล์นี้คือ "สูตรการทำงาน" ที่เราเขียนบอก Azure DevOps ว่าให้ทำอะไรบ้าง

ลองนึกภาพว่าเราจ้างพนักงานใหม่มาทำงาน
เราต้องเขียน "คู่มือการทำงาน" ให้เขาอ่านว่าต้องทำอะไรบ้าง

azure-pipelines.yml ก็คือ "คู่มือการทำงาน" ของ Pipeline นั่นเอง
บอกว่า รันเมื่อไหร่ รันบนเครื่องอะไร และทำอะไรบ้าง 
# azure-pipelines.yml อ่านทีละบรรทัด

# trigger = บอกว่าให้รัน pipeline เมื่อไหร่
# ในที่นี้คือ ทุกครั้งที่มีการ push code ขึ้น branch main
trigger:
  branches:
    include:
      - main

# pool = บอกว่าจะรันบนเครื่องอะไร
# ubuntu-latest = เครื่อง Linux ที่ Azure เตรียมให้ฟรี
# เราไม่ต้องเตรียมเครื่องเอง Azure จัดการให้
pool:
  vmImage: ubuntu-latest

# steps = บอกว่าให้ทำอะไรบ้าง เรียงตามลำดับจากบนลงล่าง
steps:

# ขั้นตอนที่ 1: ติดตั้ง Python version 3.11 บนเครื่องเสมือน
- task: UsePythonVersion@0
  inputs:
    versionSpec: '3.11'
  displayName: 'ติดตั้ง Python 3.11'

# ขั้นตอนที่ 2: ติดตั้ง library ทั้งหมดจาก requirements.txt
# และติดตั้ง browser สำหรับ Playwright
- script: |
    pip install -r requirements.txt
    playwright install chromium
  displayName: 'ติดตั้ง library ทั้งหมด'

# ขั้นตอนที่ 3: รัน test จริงๆ
- script: pytest tests/ -v
  displayName: 'รัน Playwright tests'

ภาพรวม Flow การทำงานทั้งหมด

เราเขียน test script บนเครื่องเรา (VS Code)
        ↓
เราสั่ง git push เพื่อส่ง code ขึ้น Azure Repos
        ↓
Azure Pipelines เห็นว่ามี code ใหม่เข้ามา
(เพราะ trigger ตั้งไว้ว่าให้รันทุกครั้งที่ push)
        ↓
Pipeline เปิดเครื่อง Linux เสมือนขึ้นมาเอง
ติดตั้ง Python และ library เอง
รัน test script ของเราเอง
        ↓
ถ้า test ผ่านหมด
→ แสดงสีเขียว
→ merge code ได้เลย

ถ้า test fail
→ แสดงสีแดง พร้อม error message
→ ต้องแก้ code แล้ว push ใหม่
        ↓
ดู report ได้จาก Artifacts

คำถามสัมภาษณ์ที่ควรรู้

ระดับเริ่มต้น

Q: CI/CD คืออะไร?
A: แนวคิดที่ทำให้การรัน test และ deploy เป็นอัตโนมัติ ทุกครั้งที่มีการ push code ไม่ต้องสั่งเอง ช่วยให้ bug ไม่หลุดไป production

Q: Pipeline คืออะไร?
A: ไฟล์ที่เขียนสูตรการทำงานไว้ล่วงหน้า บอกว่าเมื่อมี code ใหม่เข้ามาให้ทำอะไรบ้าง เรียงตามลำดับ

Q: CI/CD กับ Pipeline เกี่ยวกันยังไง?
A: CI/CD คือแนวคิด Pipeline คือวิธีที่ทำให้แนวคิดนั้นเกิดขึ้นจริง

Q: Repo คืออะไร ทำไมต้องมีก่อน Pipeline?
A: Repo คือที่เก็บ code บน cloud Pipeline ต้องดึง code จาก Repo มารัน ถ้าไม่มี Repo Pipeline ไม่รู้จะเอา code มาจากไหน

Q: trigger ใน azure-pipelines.yml คืออะไร?
A: บอกว่าให้รัน pipeline เมื่อไหร่ เช่น ทุกครั้งที่ push ขึ้น branch main

ระดับกลาง

Q: Branch คืออะไร ทำไมต้องมีหลาย branch?
A: Branch คือสำเนาของ code ที่แยกออกมา ทำให้แต่ละคนทำงานแยกกันได้โดยไม่กระทบกัน เช่น main สำหรับ production, develop สำหรับพัฒนา

Q: Pull Request คืออะไร?
A: การขอ merge code จาก branch นึงไปอีก branch เหมือนส่งงานให้หัวหน้าตรวจก่อน merge เมื่อเปิด PR pipeline จะรัน test อัตโนมัติ

Q: ถ้า test fail ใน pipeline ต้องทำอะไร?
A: ดู error log ใน pipeline แก้ code แล้ว push ขึ้นใหม่ pipeline จะรันอีกครั้งอัตโนมัติ

Q: Artifact ใน pipeline คืออะไร?
A: ผลลัพธ์ที่ได้จาก pipeline เช่น test report ที่ดาวน์โหลดมาดูได้หลัง pipeline รันเสร็จ
สิ่งที่ต้องมีก่อนเริ่ม:
มี account Azure DevOps — สมัครฟรีที่ dev.azure.com
มี VS Code ติดตั้งแล้ว
มี Python ติดตั้งแล้ว
มี test script เขียนไว้แล้ว (จากบทที่ 1-3)

1 สร้าง Project ใน Azure DevOps

Project คือ "พื้นที่ทำงาน" ที่รวม Repo, Pipeline, Boards ไว้ด้วยกัน เหมือนสร้างโฟลเดอร์หลักสำหรับโปรเจกต์นี้

1. เปิด https://dev.azure.com แล้ว login
2. ถ้ายังไม่มี Organization → กด "New Organization" แล้วตั้งชื่อ
3. กด "New Project" ที่มุมขวาบน
4. ตั้งชื่อ Project เช่น "QA-Automation"
5. เลือก Visibility เป็น "Private" (ไม่ให้คนนอกเห็น)
6. กด "Create Project"
เสร็จแล้ว! จะเห็นหน้า Project Overview ที่มีเมนูซ้ายมือ

2 ดู Repo ที่ถูกสร้างให้อัตโนมัติ

Azure DevOps จะสร้าง Repo ให้อัตโนมัติเมื่อสร้าง Project เราแค่ต้อง Initialize มัน

1. กดที่ "Repos" ในเมนูซ้าย
2. จะเห็น Repo ที่ชื่อเดียวกับ Project ถูกสร้างให้แล้ว
3. กด "Initialize" เพื่อสร้าง branch main
   (ถ้าไม่ Initialize จะยังไม่มี branch ให้ push ขึ้นไปได้)
4. Copy URL ของ Repo ไว้ใช้ในขั้นตอนที่ 5
   URL จะหน้าตาแบบนี้:
   https://dev.azure.com/ชื่อ-org/QA-Automation/_git/QA-Automation
เสร็จแล้ว! ตอนนี้มี Repo พร้อมรับ code แล้ว

3 เตรียม code บนเครื่องเรา

เปิด Terminal บน Mac แล้วพิมพ์ทีละบรรทัด

# สร้างโฟลเดอร์โปรเจกต์
mkdir qa-automation
cd qa-automation

# สร้าง virtual environment (กล่องแยกสำหรับโปรเจกต์นี้)
python3 -m venv venv

# เปิดใช้งาน virtual environment
source venv/bin/activate

# ติดตั้ง library ทั้งหมด
pip install pytest playwright pytest-playwright requests robotframework robotframework-browser

# ติดตั้ง browser สำหรับ Playwright
playwright install chromium

# สร้างโครงสร้างโฟลเดอร์
mkdir pages tests api_tests robot_tests

จากนั้นสร้างไฟล์ requirements.txt ใน VS Code แล้วใส่ข้อมูลนี้

pytest
playwright
pytest-playwright
requests
pytest-html
robotframework
robotframework-browser
requirements.txt สำคัญมาก! Pipeline จะใช้ไฟล์นี้ติดตั้ง library ให้อัตโนมัติบนเครื่องเสมือน

4 สร้างไฟล์ azure-pipelines.yml

สร้างไฟล์ชื่อ azure-pipelines.yml ใน root ของโปรเจกต์ (ระดับเดียวกับ requirements.txt)

ชื่อไฟล์ต้องเป็น azure-pipelines.yml เท่านั้น และต้องอยู่ใน root ของโปรเจกต์ ไม่ใช่ในโฟลเดอร์ย่อย
# azure-pipelines.yml
# ไฟล์นี้บอก Azure DevOps ว่าให้ทำอะไรเมื่อมี code ใหม่เข้ามา

# trigger = รัน pipeline เมื่อไหร่
# ในที่นี้คือทุกครั้งที่ push ขึ้น branch main หรือ develop
trigger:
  branches:
    include:
      - main
      - develop

# pool = รันบนเครื่องอะไร
# ubuntu-latest = เครื่อง Linux ที่ Azure เตรียมให้ฟรี ไม่ต้องเตรียมเอง
pool:
  vmImage: ubuntu-latest

# stages = ขั้นตอนใหญ่ๆ แต่ละ stage รันแยกกัน
stages:

# Stage 1: รัน Playwright UI Tests
- stage: PlaywrightTests
  displayName: 'Playwright UI Tests'
  jobs:
  - job: RunPlaywright
    steps:

    # ขั้นตอนที่ 1: ติดตั้ง Python
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
      displayName: 'ติดตั้ง Python 3.11'

    # ขั้นตอนที่ 2: ติดตั้ง library
    - script: |
        pip install -r requirements.txt
        playwright install chromium
      displayName: 'ติดตั้ง library'

    # ขั้นตอนที่ 3: รัน test
    - script: pytest tests/ -v --html=playwright-report.html
      displayName: 'รัน Playwright tests'

    # ขั้นตอนที่ 4: เก็บ report ไว้ให้ดาวน์โหลด
    # condition: always() = เก็บเสมอ แม้ test จะ fail
    - task: PublishBuildArtifacts@1
      condition: always()
      inputs:
        pathToPublish: 'playwright-report.html'
        artifactName: 'playwright-report'
      displayName: 'เก็บ Playwright Report'

# Stage 2: รัน API Tests
# dependsOn: PlaywrightTests = รอให้ Stage 1 เสร็จก่อน
- stage: APITests
  displayName: 'API Tests'
  dependsOn: PlaywrightTests
  jobs:
  - job: RunAPITests
    steps:
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
    - script: pip install -r requirements.txt
      displayName: 'ติดตั้ง library'
    - script: pytest api_tests/ -v
      displayName: 'รัน API tests'

# Stage 3: รัน Robot Framework Tests
- stage: RobotTests
  displayName: 'Robot Framework Tests'
  dependsOn: APITests
  jobs:
  - job: RunRobotTests
    steps:
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
    - script: |
        pip install -r requirements.txt
        rfbrowser init
      displayName: 'ติดตั้ง library'
    - script: robot robot_tests/
      displayName: 'รัน Robot tests'
    - task: PublishBuildArtifacts@1
      condition: always()
      inputs:
        pathToPublish: 'report.html'
        artifactName: 'robot-report'
      displayName: 'เก็บ Robot Report'

5 Push code ขึ้น Azure Repos

ตอนนี้เรามี code บนเครื่องแล้ว ต้องส่งขึ้น Azure Repos เพื่อให้ Pipeline ดึงไปรันได้

เปิด Terminal แล้วพิมพ์ทีละบรรทัด

# เข้าไปในโฟลเดอร์โปรเจกต์
cd qa-automation

# เริ่มต้นใช้ git ในโฟลเดอร์นี้
# git คือระบบที่ช่วยติดตามการเปลี่ยนแปลงของ code
git init

# บอก git ว่าจะ track ไฟล์ทั้งหมดในโฟลเดอร์นี้
git add .

# บันทึก snapshot ของ code พร้อมข้อความอธิบายว่าทำอะไร
git commit -m "feat: add QA automation tests and pipeline"

# เชื่อม local กับ Azure Repos
# แทนที่ URL ด้วย URL ที่ copy มาจากขั้นตอนที่ 2
git remote add origin https://dev.azure.com/your-org/QA-Automation/_git/QA-Automation

# เปลี่ยนชื่อ branch เป็น main
git branch -M main

# push code ขึ้น Azure Repos ครั้งแรก
# -u = ตั้งค่า default ว่า push ไปที่ origin/main
git push -u origin main
เสร็จแล้ว! ไปดูใน Azure DevOps → Repos จะเห็นไฟล์ทั้งหมดขึ้นไปแล้ว

6 สร้าง Pipeline ใน Azure DevOps

ตอนนี้ code อยู่บน Repos แล้ว ขั้นตอนนี้คือบอก Azure DevOps ว่าให้ใช้ไฟล์ azure-pipelines.yml ที่เราสร้างไว้

1. ใน Project กดที่ "Pipelines" ในเมนูซ้าย
2. กด "New Pipeline" ที่มุมขวาบน
3. เลือก "Azure Repos Git"
   (เพราะ code เราอยู่ใน Azure Repos)
4. เลือก Repo ที่เพิ่ง push ขึ้นไป
5. เลือก "Existing Azure Pipelines YAML file"
   (เพราะเราสร้างไฟล์ yml ไว้แล้ว)
6. เลือก Branch: main
7. เลือก Path: /azure-pipelines.yml
8. กด "Continue"
9. จะเห็นไฟล์ yml ที่เราเขียนไว้แสดงขึ้นมา
10. กด "Run" เพื่อรัน pipeline ครั้งแรก
Pipeline ถูกสร้างแล้ว และกำลังรันครั้งแรก!

7 ดู Pipeline รัน

หลังกด Run จะเห็นหน้า Pipeline รันอยู่ มาดูว่าแต่ละส่วนคืออะไร

หน้า Pipeline จะแสดง:

Stages:
  Playwright UI Tests  → กำลังรัน หรือ ผ่าน หรือ fail
  API Tests            → รอ Stage แรกเสร็จก่อน
  Robot Framework Tests → รอ Stage สองเสร็จก่อน

วิธีดู log:
1. คลิกที่ Stage ที่ต้องการดู
2. คลิกที่ Job
3. จะเห็น log ทุก step ที่รัน
4. ถ้า fail จะเห็น error message สีแดง

วิธีดาวน์โหลด Report:
1. หลัง pipeline รันเสร็จ
2. กดที่ "Artifacts" ด้านขวาบน
3. จะเห็น playwright-report และ robot-report
4. กดดาวน์โหลดได้เลย
เสร็จแล้ว! ทุกครั้งที่ push code ขึ้น Azure Repos Pipeline จะรัน test อัตโนมัติทันที ไม่ต้องสั่งเอง

ทุกครั้งที่แก้ code หรือเพิ่ม test ใหม่ ทำแค่นี้

# บันทึกการเปลี่ยนแปลง
git add .

# เขียนอธิบายว่าแก้อะไร
git commit -m "feat: เพิ่ม test case login"

# ส่งขึ้น Azure Repos
git push

# Pipeline จะรัน test อัตโนมัติทันที ไม่ต้องทำอะไรเพิ่ม

ถ้า Pipeline Fail ต้องทำอะไร?

Error ที่เห็นสาเหตุวิธีแก้
ModuleNotFoundError: No module named 'xxx'ลืมใส่ library ใน requirements.txtเพิ่ม library ใน requirements.txt แล้ว push ใหม่
playwright: command not foundลืมรัน playwright install chromium ใน pipelineเพิ่ม playwright install chromium ใน azure-pipelines.yml
FAILED tests/test_login.pytest case fail จริงๆแก้ code หรือแก้ test แล้ว push ใหม่
YAML syntax errorเขียน azure-pipelines.yml ผิด indentationเช็ค indentation ใน yml ให้ถูกต้อง (ใช้ space ไม่ใช่ tab)
จำไว้เสมอ:
azure-pipelines.yml = สูตรที่บอก Pipeline ว่าให้ทำอะไร
Repo = ที่เก็บ code และไฟล์ yml
Pipeline = ตัวที่อ่าน yml แล้วทำตาม

ทั้ง 3 อย่างนี้ต้องมีครบถึงจะทำงานได้
