ถ้าทีมยังไม่เคยทำ Automate เลย จะเริ่มยังไง?

จะเริ่มจาก API Automation ก่อนเสมอ เพราะ

เหตุผลที่เริ่ม API ก่อน UI:

1. เร็วกว่า — ไม่ต้องเปิด browser ไม่ต้องรอหน้าเว็บโหลด
2. stable กว่า — ไม่ขึ้นกับ UI ที่เปลี่ยนบ่อย
3. ครอบคลุมกว่า — เช็คได้ทุก endpoint ทุก case
4. ถ้า API พัง UI ก็พังด้วย เพราะ UI ดึงข้อมูลจาก API
เพราะฉะนั้นแก้ API ให้ถูกก่อน แล้วค่อยทำ UI

Tech Stack ที่ใช้

Toolทำอะไรใช้เมื่อไหร่
Pythonภาษาที่ใช้รัน Robotต้องมีเสมอ
Robot Frameworkตัว framework หลักต้องมีเสมอ
RequestsLibraryยิง HTTP request สำหรับ API testตอนทำ API test
Browser Libraryควบคุม browser สำหรับ UI test (รุ่นใหม่)ตอนทำ UI test
SeleniumLibraryควบคุม browser สำหรับ UI test (รุ่นเก่า)ถ้ามี codebase เก่า
ทำไมมี Library ให้เลือกหลายตัว?

Robot Framework เป็นแค่ "กรอบ" ไม่ได้ทำอะไรเอง ต้องพึ่ง Library ที่ทำงานจริงๆ

เหมือนรถยนต์ที่ต้องเลือกเครื่องยนต์:
Robot Framework = ตัวรถ
RequestsLibrary = เครื่องยนต์สำหรับยิง API
Browser Library = เครื่องยนต์สำหรับควบคุม browser (รุ่นใหม่ แนะนำ)
SeleniumLibrary = เครื่องยนต์สำหรับควบคุม browser (รุ่นเก่า)

# ============================================================
# tests/test_api.py — ทดสอบ API ด้วย Python + requests
# ============================================================

# ===== ส่วนที่ 1: Import library =====

# requests = library สำหรับยิง HTTP request
# เหมือน Postman แต่เป็น code ทำงานอัตโนมัติได้
import requests

# BASE_URL = URL หลักของ API ที่จะทดสอบ
# เก็บไว้ที่เดียว ถ้าต้องเปลี่ยนแก้แค่นี่ที่เดียว
BASE_URL = "https://jsonplaceholder.typicode.com"


# ===== ส่วนที่ 2: Test Cases =====
# ทุก function ที่ขึ้นต้นด้วย test_ จะถูก pytest รันเป็น test case
# pytest จะหา function ที่ขึ้นต้นด้วย test_ ให้อัตโนมัติ


# Test 1: เช็คว่า GET user แล้วได้ status 200 มั้ย
def test_get_user_returns_200():
    # requests.get = ส่ง GET request ไปที่ URL นี้
    # f"{BASE_URL}/users/1" = https://jsonplaceholder.typicode.com/users/1
    response = requests.get(f"{BASE_URL}/users/1")

    # assert = เช็คเงื่อนไข
    # ถ้าเงื่อนไขเป็นจริง → ผ่าน อ่านบรรทัดต่อไป
    # ถ้าเงื่อนไขเป็นเท็จ → test fail ทันที แสดง error
    # response.status_code = ตัวเลข status ที่ได้กลับมา เช่น 200, 404
    assert response.status_code == 200


# Test 2: เช็คว่า response มี field ครบมั้ย
def test_get_user_has_correct_fields():
    response = requests.get(f"{BASE_URL}/users/1")

    # .json() = แปลง response จาก text ธรรมดา → เป็น dictionary Python
    # เหมือนข้อมูล JSON ที่เห็นใน Postman panel ด้านล่าง
    body = response.json()

    # "id" in body = เช็คว่า key "id" มีอยู่ใน dictionary มั้ย
    assert "id" in body      # ต้องมี field id
    assert "name" in body    # ต้องมี field name
    assert "email" in body   # ต้องมี field email


# Test 3: เช็คว่า POST สร้างข้อมูลใหม่ได้มั้ย
def test_create_post_returns_201():
    # payload = ข้อมูลที่จะส่งไปสร้างใหม่
    # เหมือนกับที่เราใส่ใน Body tab ของ Postman
    payload = {
        "title": "test post",   # หัวข้อของ post
        "body": "hello world",  # เนื้อหาของ post
        "userId": 1             # id ของ user ที่สร้าง post
    }

    # requests.post = ส่ง POST request พร้อมข้อมูล
    # json=payload = ส่ง payload เป็น JSON body
    response = requests.post(f"{BASE_URL}/posts", json=payload)

    # 201 = Created = สร้างสำเร็จ
    assert response.status_code == 201

    # เช็คว่าข้อมูลที่ได้กลับมาตรงกับที่ส่งไปมั้ย
    assert response.json()["title"] == "test post"

# ============================================================
# pages/login_page.py — แผนที่ของหน้า login
# ไฟล์นี้บอกว่าหน้า login มีอะไรอยู่ที่ไหน
# และทำอะไรได้บ้าง
# ============================================================

class LoginPage:
    # class = พิมพ์เขียวที่บอกว่าหน้า login มีอะไรบ้าง

    # __init__ = constructor รันทันทีตอนสร้าง LoginPage ขึ้นมา
    # self = ตัวแทนของ object นี้เอง เหมือน "ตัวเอง" ในภาษาไทย
    # page = browser page ที่ Playwright ส่งมาให้
    def __init__(self, page):
        self.page = page

        # locator = วิธีบอก Playwright ว่า element อยู่ที่ไหนบนหน้าเว็บ
        # หาได้จากการ คลิกขวา → Inspect บนหน้าเว็บ แล้วดู HTML

        # "#user-name" = หา element ที่มี id="user-name"
        self.username_input = page.locator("#user-name")

        # "#password" = หา element ที่มี id="password"
        self.password_input = page.locator("#password")

        # "#login-button" = หาปุ่ม login ที่มี id="login-button"
        self.login_button = page.locator("#login-button")

        # "[data-test='error']" = หากล่องข้อความ error
        # ที่จะโชว์ตอน login ผิด
        self.error_message = page.locator("[data-test='error']")

    # method สำหรับเปิดหน้าเว็บ
    def goto(self):
        # goto = สั่งให้ browser เปิด URL นี้
        self.page.goto("https://www.saucedemo.com")

    # method สำหรับกรอก username, password แล้วกด login
    # รับค่า username และ password มาเป็น parameter
    def login(self, username, password):
        # fill = กรอกข้อความลงใน input field
        self.username_input.fill(username)
        self.password_input.fill(password)

        # click = กดปุ่ม
        self.login_button.click()

# ============================================================
# tests/test_login.py — test case สำหรับหน้า login
# ============================================================

# ===== Import library =====

# Page = type ของ browser page จาก Playwright
# expect = ตัวช่วยเช็คผลแบบ Playwright
from playwright.sync_api import Page, expect

# import LoginPage ที่เราสร้างไว้ใน pages/login_page.py
from pages.login_page import LoginPage


# ===== Test Cases =====
# ทุก function ที่ขึ้นต้นด้วย test_ จะถูก pytest รัน
# page: Page = Playwright จะสร้าง browser page ให้อัตโนมัติ


# Test 1: login ด้วยข้อมูลถูกต้อง ต้องเข้าหน้า inventory ได้
def test_login_success(page: Page):
    # สร้าง LoginPage object โดยส่ง page เข้าไป
    # เหมือนบอกว่า "ใช้แผนที่หน้า login กับ browser page นี้"
    login = LoginPage(page)

    # เปิดหน้าเว็บ
    login.goto()

    # กรอก username/password ที่ถูกต้อง แล้วกด login
    login.login("standard_user", "secret_sauce")

    # expect = เช็คว่าผลเป็นไปตามที่คาดมั้ย
    # to_have_url = URL ต้องเปลี่ยนไปเป็นหน้า inventory

คำถามสัมภาษณ์ที่ควรถาม และวิธีวิเคราะห์คำตอบ

คำถามที่ 1: ถ้าบริษัทยังไม่เคยทำ automate เลย คุณจะเริ่มยังไง?

คำตอบที่แสดงว่ารู้จริง:
→ บอกว่าจะเริ่มจาก API test ก่อน เพราะเร็วกว่า stable กว่า
→ พูดถึงการเช็ค element ว่า stable มั้ย มี id หรือ data-test attribute มั้ย
→ บอกว่าจะเริ่มจาก critical flow ก่อน เช่น login, หน้าหลัก
→ พูดถึง CI/CD ด้วยว่าหลังเขียน script เสร็จจะ push ขึ้น pipeline
คำตอบที่น่าเป็นห่วง:
→ บอกแค่ว่าจะเขียน script เลย ไม่พูดถึงการวางแผน
→ ไม่รู้ว่าต้องคุยกับ developer เรื่องอะไรก่อน
→ ไม่รู้ว่าทำไมต้องเริ่มจาก API ก่อน UI

คำถามที่ 2: คุณจะเลือกใช้ Robot Framework หรือ Playwright และทำไม?

คำตอบที่แสดงว่ารู้จริง:
→ ไม่ตอบตรงๆ ว่าอันไหนดีกว่า แต่บอกว่าขึ้นอยู่กับทีมและโปรเจกต์
→ อธิบาย trade-off ได้ เช่น Robot อ่านง่ายกว่า เหมาะทีมที่ไม่ถนัด code
→ Playwright ยืดหยุ่นกว่า เหมาะทีมที่ถนัด Python
→ ถามกลับว่าทีมถนัด code มั้ย
คำตอบที่น่าเป็นห่วง:
→ ตอบแค่ว่า Playwright ดีกว่าเพราะเร็วกว่า โดยไม่รู้ว่าทำไม
→ ไม่รู้ว่าเลือกใช้เมื่อไหร่ขึ้นอยู่กับ context อะไร

คำถามที่ 3: API test กับ UI test ต่างกันยังไง และควรทำอันไหนก่อน?

คำตอบที่แสดงว่ารู้จริง:
→ อธิบายได้ว่า API test เช็ค backend UI test เช็คหน้าเว็บที่ user เห็น
→ บอกว่าควรทำ API ก่อน เพราะเร็วกว่า ครอบคลุมกว่า
→ บอกสัดส่วนที่ดีได้ เช่น API 70% UI 30%
คำตอบที่น่าเป็นห่วง:
→ ไม่รู้ว่าต่างกันยังไง
→ ตอบแค่ว่า UI test เพราะเห็นภาพกว่า โดยไม่รู้ว่าทำไมต้องทำ API ก่อน

คำถามที่ 4: Flaky Test คืออะไร เคยเจอมั้ย แก้ยังไง?

คำตอบที่แสดงว่ารู้จริง:
→ อธิบายได้ว่า flaky test คือ test ที่บางครั้งผ่าน บางครั้งไม่ผ่าน โดยที่ code ไม่ได้เปลี่ยน
→ บอกสาเหตุได้ เช่น locator ไม่ stable เพราะ UI เปลี่ยนบ่อย network ช้า
→ บอกวิธีแก้ได้ เช่น ใช้ data-test attribute ใช้ Auto-wait ของ Playwright
คำตอบที่น่าเป็นห่วง:
→ ไม่รู้ว่า flaky test คืออะไร
→ ไม่เคยเจอปัญหานี้เลย แสดงว่าอาจไม่ได้ทำ automate จริงๆ

คำถามที่ 5: Page Object Model คืออะไร ทำไมต้องใช้?

คำตอบที่แสดงว่ารู้จริง:
→ อธิบายได้ว่าคือการแยก locator ออกจาก test case
→ บอกได้ว่าทำไมต้องแยก เช่น ถ้า developer เปลี่ยน id แก้แค่ที่เดียว ไม่ต้องแก้ทุก test case
→ ยกตัวอย่างได้ว่าถ้าไม่ใช้จะเกิดอะไร
คำตอบที่น่าเป็นห่วง:
→ ไม่รู้ว่า Page Object Model คืออะไร
→ รู้แค่ชื่อ แต่อธิบายไม่ได้ว่าทำไมต้องใช้

สรุปสิ่งที่ต้องฟังจาก Candidate

รู้จริงรู้แค่ผิวเผิน
อธิบาย process ได้ตั้งแต่ต้นจนจบ ตอบได้แค่ทฤษฎี แต่อธิบาย process ไม่ได้
รู้ว่าต้องเริ่มจากไหน ทำไม ไม่รู้ว่าจะเริ่มยังไงถ้าไม่มีอะไรเลย
รู้ trade-off ของแต่ละ tool ไม่รู้ว่า flaky test คืออะไร
พูดถึงปัญหาจริงๆ ที่จะเจอ เช่น flaky test ไม่รู้ว่า CI/CD เกี่ยวกับ automate ยังไง
รู้ว่าต้องทำงานร่วมกับ developer ยังไง บอกแค่ว่าใช้ AI ช่วยเขียน โดยไม่เข้าใจว่า code ทำงานยังไง
พูดถึง CI/CD ได้ ตอบแค่ว่าอันนี้ดีกว่าอันนั้น โดยไม่รู้ว่าทำไม

CI/CD กับ Azure DevOps คืออะไร?

CI/CD = แนวคิด ที่ทำให้การรัน test เป็นอัตโนมัติทุกครั้งที่มีการ push code

Azure DevOps = platform ที่ทำให้ CI/CD เกิดขึ้นจริง มีทั้ง Repos (เก็บ code) และ Pipelines (รัน test อัตโนมัติ)

Pipeline = ไฟล์ ที่เราเขียนบอกว่าให้ทำอะไรบ้าง เมื่อมี code ใหม่เข้ามา

ทำไมต้องมี CI/CD?

ไม่มี CI/CDมี CI/CD
QA ต้องนั่งรัน test เองทุกครั้งระบบรัน test อัตโนมัติทันทีที่มีการ push code
ถ้าลืมรัน bug หลุดไป productionถ้า test fail แจ้งเตือนทันที merge ไม่ได้
ไม่รู้ว่า code ใหม่ทำให้อะไรพังรู้ทันทีว่า test ไหน fail เพราะอะไร

ไฟล์ azure-pipelines.yml ที่ต้องมีใน Repo

# azure-pipelines.yml
# ไฟล์นี้ต้องอยู่ใน root ของโปรเจกต์ ระดับเดียวกับ requirements.txt

# trigger = รัน pipeline เมื่อไหร่
# ทุกครั้งที่ push ขึ้น branch main หรือ develop
trigger:
  branches:
    include:
      - main
      - develop

# pool = รันบนเครื่องอะไร
# ubuntu-latest = เครื่อง Linux ที่ Azure เตรียมให้ฟรี
pool:
  vmImage: ubuntu-latest

# stages = ขั้นตอนใหญ่ๆ แต่ละ stage รันแยกกัน
stages:

# Stage 1: รัน Playwright UI Tests
- stage: PlaywrightTests
  displayName: 'Playwright UI Tests'
  jobs:
  - job: RunPlaywright
    steps:
    # ติดตั้ง Python version 3.11
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
    # ติดตั้ง library ทั้งหมดจาก requirements.txt
    - script: |
        pip install -r requirements.txt
        playwright install chromium
      displayName: 'ติดตั้ง library'
    # รัน test จริงๆ
    - script: pytest tests/ -v --html=playwright-report.html
      displayName: 'รัน Playwright tests'
    # เก็บ report ไว้ให้ดาวน์โหลด
    - task: PublishBuildArtifacts@1
      condition: always()
      inputs:
        pathToPublish: 'playwright-report.html'
        artifactName: 'playwright-report'

# Stage 2: รัน API Tests
- stage: APITests
  displayName: 'API Tests'
  dependsOn: PlaywrightTests
  jobs:
  - job: RunAPITests
    steps:
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
    - script: pip install -r requirements.txt
    - script: pytest api_tests/ -v

# Stage 3: รัน Robot Framework Tests
- stage: RobotTests
  displayName: 'Robot Framework Tests'
  dependsOn: APITests
  jobs:
  - job: RunRobotTests
    steps:
    - task: UsePythonVersion@0
      inputs:
        versionSpec: '3.11'
    - script: |
        pip install -r requirements.txt
        rfbrowser init
    - script: robot robot_tests/
    - task: PublishBuildArtifacts@1
      condition: always()
      inputs:
        pathToPublish: 'report.html'
        artifactName: 'robot-report'

TypeScript คือ JavaScript ที่ Microsoft เพิ่มระบบ "ตรวจสอบความถูกต้อง" เข้ามา

JavaScript = พนักงานที่ทำงานได้ แต่ไม่มีใครตรวจสอบ ทำผิดก็รู้ตอน run แล้ว

TypeScript = พนักงานที่มีหัวหน้าคอยตรวจงานก่อนส่ง ทำผิดรู้ทันทีก่อน run เลย

ตัวอย่างให้เห็นภาพชัดๆ

สมมุติเราเขียนฟังก์ชันบวกเลข

JavaScript — ไม่มีการตรวจสอบ

function add(a, b) {
    return a + b
}

add(5, 10)        // ได้ 15 ถูกต้อง
add(5, "hello")   // ได้ "5hello" ผิดแต่ไม่มี error!
                  // รู้ว่าผิดตอน run แล้วเท่านั้น

TypeScript — มีการตรวจสอบก่อน run

function add(a: number, b: number): number {
    return a + b
}
//          ↑              ↑         ↑
//     a ต้องเป็น number  b ต้องเป็น number  ผลลัพธ์ต้องเป็น number

add(5, 10)        // ได้ 15 ถูกต้อง
add(5, "hello")   // Error ทันทีก่อน run!
                  // TypeScript บอกว่า "hello" ไม่ใช่ number

let user: { name: string, age: number } = {
